### 알게 된 것 / 삽질

- **PostgREST `.or()` 의 nested FK 한계**: `breweries.name_ko.ilike.%X%` 가 일반 OR syntax
  안에 못 들어감 — `referencedTable` 옵션이 OR 시점엔 1-table only. 우회: brewery 매칭은
  별도 쿼리로 brewery_id 풀 만들고 main 쿼리는 `brewery_id.in.(...)` 로 합침. 이 패턴
  앞으로 검색 추가 시 (블로그·체크인 등) 재사용.
- **PostgREST query string layer 의 SQL injection 과 별개 escape**: Supabase JS 가 SQL
  injection 자체는 막지만 query string parser 는 별개. `,` `(` `)` 가 사용자 input 에 들어가면
  parser 깨짐. `sanitizeSearch` 로 escape 해야 함.
- **`as const` literal narrowing 의 ko/en 미러 트릭**: ko 가 `as const` 면 string literal
  까지 좁혀져서 en 의 다른 문자열이 type 안 맞음. recursive `Widen<T>` 로 leaf 만 string 으로
  widen + 중첩 object 그대로. DotPath 추출도 여전히 정확. (5-07 에 깔아놓은 패턴이 5-08 의
  대규모 dictionary 확장에도 그대로 작동.)
- **DotPath 와 동적 한국어 key**: `t(locale, "categories.탁주")` 는 type-safe 하지만
  `t(locale, \`categories.${product.category}\` as TKey)` 는 type 우회. 그래서 dynamic 키는
  `tCategory` / `tCategoryHint` / `tTaste` / `tRegion` 4 helper 별도 (signature: `(locale,
  koValue: string) => string`). 호출부에서 type 우회를 노출하지 않음.
- **Async server component vs client prop drilling**: SSR locale 이 server tree 어디서나
  자유로워서 TasteRadar / PhotoPlaceholder 같은 leaf 에서도 자체 fetch 가능. client
  component 만 부모가 prop 으로 주입. 처음에는 모두 prop drilling 하다가 검증 단계에서 async
  server 패턴이 깔끔하다고 깨달음.
- **fixed list 매핑 (지역) vs 문자열 데이터 매핑 (ingredients/pairing) 의 경계**: 17 시도는
  fixed dictionary 로 끝, 791 product 의 가변 ingredient/pairing 은 AI 번역 + DB 칼럼 분리
  필요. UI 라벨 i18n 으로 못 풀고 데이터 i18n 으로 가야 함 (다음 세션).
- **더술닷컴 visitingBrewery 페이지 = JS rendering 없는 정적 HTML**: `recordCountPerPage`
  query param 안 먹음 (5개씩 고정), pageIndex 만 작동. 페이지 1~10 fetch 해서 unique 39곳
  모음. WebFetch 로 충분 (playwright 불필요).
- **카카오 비즈 인증 5-07 부터 미정 그대로**: Kim 이 신청 안 했고, 코드 (signInWithKakao
  주석 블록) 는 LoginForm 안에 보존 — 비즈 인증 통과 후 주석만 풀면 5분 내 부활.
